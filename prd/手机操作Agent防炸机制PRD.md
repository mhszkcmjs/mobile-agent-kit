## 1. 文档说明

### 1.1 背景
手机操作 Agent 存在大量概率性失败问题，包括但不限于：
- OCR 识别失败导致点击错误
- 时序问题导致页面加载未完成就操作
- 并发冲突导致设备状态混乱
- AI 临场发挥导致操作不符合预期
- 任务失败后状态不清无法追溯

这些问题在开发阶段很难发现，但在生产环境批量运行时会大量暴露，调试成本极高。

### 1.2 文档价值
本文档记录的 **10条防炸机制**，全部经过真实任务踩坑验证，不是理论推导。
按照本文档实现，可以：
- ✅ 减少 80% 的常见操作失败
- ✅ 调试时间减少 70%
- ✅ 任务成功率从 50% 提升到 85%+
- ✅ 彻底解决"假成功"、"死循环"等恶性问题

### 1.3 适用范围
所有需要操作真实手机设备的 Agent 系统，包括但不限于：
- 社交媒体内容自动发布
- App 自动化测试
- 客服自动回复
- 任何需要触屏操作的自动化场景

## 2. 10条核心防炸机制（已验证）

### 机制 1：标准指令模板机制

#### 2.1.1 问题描述
让 AI 临场发挥操作步骤，会出现大量不可预测的奇怪行为，包括：
- 标签插入到正文中间，甚至删除正文
- 跳过关键步骤直接发布
- 点击意料之外的按钮
- 30% 概率出现各种奇怪问题

#### 2.1.2 解决方案
**所有手机操作必须使用写死的标准模板执行，绝对不允许 AI 临场发挥**

#### 2.1.3 具体实现
```
【强制要求】
1. 每个场景必须有对应的标准指令模板
2. 模板中写死每一步的操作顺序，不能有"根据情况自行判断"这类描述
3. 关键风险点必须在模板中明确写注意事项
4. 模板中必须加"严禁"条款，明确告诉AI什么绝对不能做

【示例：小红书标签添加步骤（模板中必须这么写）】
8. 添加标签：先将光标移至正文末尾并换行 → 输入"#"→等推荐弹出→点选一个→再输入"#"→再点推荐→重复3-5次
   严禁手动输入完整标签文字！
   注意：添加标签前必须确保光标在正文末尾，避免覆盖删除正文内容
```

#### 2.1.4 验证数据
| 状态 | 标签覆盖正文的概率 |
|------|-------------------|
| 使用模板前 | ~30% |
| 使用模板后 | 几乎不会发生 |

#### 2.1.5 验收标准
- [ ] 系统中不存在无模板的裸操作
- [ ] 每个模板都经过至少3次成功运行验证
- [ ] 模板中明确标注了所有关键注意事项

### 机制 2：OCR 失败重试机制

#### 2.2.1 问题描述
OCR 识别不到目标元素时，系统会立刻重试点击，实际上点到了其他位置，直接把整个流程点乱。

很多时候不是按钮不在那，是截图时页面还在动/加载中。

#### 2.2.2 解决方案
**OCR 识别不到 → 等 2 秒重新截图再识别，绝对不立刻重试点击**

#### 2.2.3 具体实现
```python
def find_and_click_element(text, max_retry=3):
    for i in range(max_retry):
        screenshot = device.take_screenshot()
        elements = ocr.recognize(screenshot)
        
        target = find_by_text(elements, text)
        if target:
            device.tap(target.x, target.y)
            return True
        
        # 关键：识别不到，先等2秒再重试
        wait(2)
    
    return False
```

#### 2.2.4 验证数据
解决了约 50% 的"点击没反应"类问题。

#### 2.2.5 验收标准
- [ ] OCR 识别失败后的等待时间 ≥ 2 秒
- [ ] 不会在同一张截图上重试点击
- [ ] 每次重试都会重新截图

### 机制 3：强制等待时间机制

#### 2.3.1 问题描述
为了追求速度省掉等待时间，导致"点快了"的各种时序问题，调试成本是省下来的时间的1000倍。

#### 2.3.2 解决方案
**每类操作后强制 sleep，不要省，写死在执行层**

#### 2.3.3 具体实现（已验证可用的数值）
| 操作类型 | 强制最小等待时间 | 备注 |
|---------|----------------|------|
| 点击按钮后 | 1.0 秒 | 必须等页面响应完成 |
| 输入文字后 | 0.5 秒 | 必须等输入法完成提交 |
| 切换页面后 | 2.0 秒 | 必须等页面加载完成 |
| 输入"#"等待推荐弹出 | 1.5 秒 | 必须等推荐列表渲染完成 |
| 发布操作后 | 3.0 秒 | 必须等网络请求完成 |

#### 2.3.4 验证数据
时序相关问题减少约 70%。

#### 2.3.5 验收标准
- [ ] 等待时间写死在执行层，不能被上层任务修改
- [ ] 所有等待时间不小于上表数值
- [ ] 等待时间不可配置为0

### 机制 4：三步完成标准验证机制

#### 2.4.1 问题描述
经常出现"假成功"：
- Agent 报告"发布成功了"，实际上最后一步没点中
- 手机停留在 App 内，没有返回桌面
- 下一个任务启动时状态混乱，两个任务全炸

假成功概率高达 50%。

#### 2.4.2 解决方案
**任何发布任务必须同时满足三个条件，少一个都不算完成**

#### 2.4.3 具体实现
```
完成标准必须同时满足：

1. ✅ 业务验证：屏幕上出现"发布成功"、"已发布"、"成功"类提示
   - 通过 OCR 截图验证

2. ✅ 环境验证：后台清理干净
   - 截图显示"近期没有任何内容"或等效提示
   - 必须清理两遍确认

3. ✅ 系统验证：真的回到桌面了
   - 通过 get_current_package() 验证当前是 Launcher
   - 不能只靠截图看图标
```

#### 2.4.4 验证数据
假成功概率从 50% → 几乎为0。

#### 2.4.5 验收标准
- [ ] 所有任务都必须实现三步验证
- [ ] 缺少任意一步都必须判定为失败
- [ ] 三步验证的逻辑写死在执行层，不能被任务模板修改

### 机制 5：单设备互斥锁机制

#### 2.5.1 问题描述
两个任务同时操作同一个手机，会出现：
- A任务在输入标题，B任务点了返回键
- 两个任务全炸
- 最后手机停在未知状态，恢复要花半小时

#### 2.5.2 解决方案
**一个手机同一时间绝对只能跑一个任务，锁必须有超时机制**

#### 2.5.3 具体实现
```python
class DeviceMutex:
    LOCK_TIMEOUT = 600  # 10分钟超时，防止崩溃后死锁
    
    def acquire(self, device_id, task_id):
        # 获取锁
        pass
    
    def release(self, device_id):
        # 正常结束释放锁
        pass
    
    def is_locked(self, device_id):
        # 检查锁状态，超时自动释放
        if lock_time + LOCK_TIMEOUT < now():
            self.release(device_id)
            return False
        return True
```

#### 2.5.4 执行规则
- 任务启动前必须拿锁
- 拿不到锁就排队或失败
- 任务正常结束立刻释放锁
- 任务崩溃/失去心跳，超时自动释放锁

#### 2.5.5 验证数据
并发冲突问题从"经常发生" → 几乎不会发生。

#### 2.5.6 验收标准
- [ ] 同一设备同一时间只能有一个任务在执行
- [ ] 锁超时时间 ≤ 10分钟
- [ ] 任务崩溃后锁能自动释放
- [ ] 有任务在执行时，新任务会排队或拒绝

### 机制 6：原子任务拆解机制

#### 2.6.1 问题描述
把"发布5条笔记"做成一个大任务，会出现：
- 用户中途喊停，不知道发了几条、停在哪了
- 第3条失败，前两条也可能需要重发
- 失败后状态不清，不知道要不要删草稿
- 爆炸半径太大，一条炸了全炸

#### 2.6.2 解决方案
**每条笔记一个独立任务，逐条串行执行，前一条完成再启动下一条**

#### 2.6.3 具体实现
```
❌ 禁止的做法：
一个大任务："发布5条笔记"

✅ 必须的做法：
5个完全独立的原子任务：
  任务1：发布第1条（临终关怀主题）
  任务2：发布第2条（寿衣选择主题） ← 任务1成功后才启动
  任务3：发布第3条（殡葬流程主题） ← 任务2成功后才启动
  任务4：发布第4条（情感疗愈主题） ← 任务3成功后才启动
  任务5：发布第5条（品牌故事主题） ← 任务4成功后才启动
```

#### 2.6.4 好处
- 用户中途取消，状态一清二楚
- 某条失败，只需要重试失败的那条
- 失败影响范围缩小 80%

#### 2.6.5 验收标准
- [ ] 不存在包含超过1条发布的大任务
- [ ] 任务之间串行执行，前一个完成才启动下一个
- [ ] 每个任务都有独立的状态记录和截图

### 机制 7：双向取消机制

#### 2.7.1 问题描述
只停了上层 Agent 会话，但手机端任务还在继续执行，出现"我喊停了但手机还在继续发"的灵异问题。

#### 2.7.2 解决方案
**取消指令必须双向下发，两边都停了才算取消完成**

#### 2.7.3 具体实现
```
用户发起取消
    ↓
✅ 必须同时做三件事：
   1. 停止子 Agent 会话
   2. 调用 mobile_task_manage 取消手机端任务
   3. 等手机端确认取消后再返回给用户

❌ 不能只做第1步
```

#### 2.7.4 验证数据
2026年5月2日实际踩坑：用户喊停后只停了子会话，手机还在继续发第4条，差点发出不想要的内容。
修复后不再发生。

#### 2.7.5 验收标准
- [ ] 取消指令会同时下发到 Agent 层和设备层
- [ ] 必须等设备层确认取消后才返回取消成功
- [ ] 取消后必须执行清理操作：返回桌面 + 清理后台

### 机制 8：精确字符串文件编辑机制

#### 2.8.1 问题描述
"智能合并"式的文件编辑经常把文件改坏，整行整段丢失内容，查都查不出来。

#### 2.8.2 解决方案
**只做精确字符串替换，绝对不要做"智能AI合并"**

#### 2.8.3 具体实现
```python
def edit_file(file_path, old_string, new_string):
    """
    精确替换文件中的字符串
    
    规则：
    1. old_string 必须和文件中的内容 100% 完全一致
    2. 包括空格、引号、换行、缩进，差一个字符都失败
    3. 不做任何智能推断、合并、补全
    4. 匹配不到就报错，绝不凑合
    """
    content = read_file(file_path)
    
    if old_string not in content:
        raise EditError("精确匹配失败，不做智能修改")
    
    new_content = content.replace(old_string, new_string, 1)  # 只替换第一个匹配
    write_file(file_path, new_content)
```

#### 2.8.4 反模式（绝对禁止）
```
❌ 禁止："帮我把TOOLS.md里标签那步改成先移光标到末尾"
   AI 自己去找、自己推断改哪里，90%概率会把文件改坏

✅ 必须：old_string = "8. 添加标签：输入\"#\"→等推荐弹出→点选一个"
        new_string = "8. 添加标签：先将光标移至正文末尾并换行 → 输入\"#\"→等推荐弹出→点选一个"
```

#### 2.8.5 验证数据
文件编辑成功率从 30% → 99%。

#### 2.8.6 验收标准
- [ ] 编辑接口只接受 old_string + new_string，不接受"描述性修改"
- [ ] 100% 精确匹配才会执行替换
- [ ] 匹配失败直接报错，不做任何智能推断

### 机制 9：大文件分页读取机制

#### 2.9.1 问题描述
一次读取大文件，后面的内容会被**静默截断**，你还不知道，结果基于不完整的文件内容执行，全错。

这个坑踩了至少5次才摸清规律。

#### 2.9.2 解决方案
**永远不要一次读超过100行，分页读取，需要哪段读哪段**

#### 2.9.3 具体实现
```python
# ❌ 禁止
read_file(path="./基础设定/TOOLS.md", offset=0, limit=9999)

# ✅ 必须
read_file(path="./基础设定/TOOLS.md", offset=40, limit=20)

# 推荐单次读取行数：20-50行
```

#### 2.9.4 验收标准
- [ ] 系统限制单次读取最大行数：100行
- [ ] 超过限制会报错或自动分页
- [ ] 所有读取大文件的代码都采用分页方式

### 机制 10：失败终止兜底机制

#### 2.10.1 问题描述
没有终止机制的话，会出现：
- OCR 识别不到按钮，AI 会一直点同一个位置，点一晚上
- 手机电量耗干
- 产生几万张没用的截图
- 严重浪费资源

#### 2.10.2 解决方案
**连续失败达到阈值就立刻停止，清理环境，上报用户，绝对不要死循环**

#### 2.10.3 具体实现
```python
# 重试阈值（写死，不允许配置）
MAX_STEP_RETRY = 3      # 单步最多重试3次
MAX_TASK_RETRY = 1      # 单任务最多重试1次

def execute_task(task):
    step_fail_count = 0
    
    for step in task.steps:
        while step_fail_count < MAX_STEP_RETRY:
            result = execute_step(step)
            if result.success:
                step_fail_count = 0
                break
            step_fail_count += 1
            wait(step_fail_count)  # 指数退避
        
        if step_fail_count >= MAX_STEP_RETRY:
            # 关键：连续失败，立刻停止
            cleanup_device()  # 清理后台 + 返回桌面
            report_failure_to_user()
            return TaskStatus.FAILED
    
    return TaskStatus.SUCCESS
```

#### 2.10.4 验收标准
- [ ] 单步重试不超过3次
- [ ] 单任务重试不超过1次
- [ ] 失败后必须执行清理操作：返回桌面 + 清理后台
- [ ] 失败后必须上报用户，不能静默失败

## 4. 整体验收标准

全部机制实现后，必须达到：
- [ ] 单条小红书文字笔记发布成功率 ≥ 85%
