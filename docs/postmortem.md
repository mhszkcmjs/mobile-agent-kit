# 手机自动化 Agent

## 开篇

我花了点时间做一个能自主操作手机的 AI agent，目标是让它替我自动发小红书笔记、刷推送把内容总结回来。最后我决定不在这个方向继续投入。这篇文章想讲清楚为什么。

简单交代下背景。自己确实有重复劳动需求（手上几个小红书账号每天都要发，整套点点点的流程做一次要 30 秒到 1 分钟），动机很自然就有了。不是要做什么改变世界的产品，就是想做个能日常用、能省时间的工具。

中间我经历了从"想做手机 AI 助理"到"想做 Mac AI 助理"再到"判断 GUI Agent 这条路有结构性问题"的认知转变。这一篇主要想分享：项目做到了什么程度、卡在了哪里、为什么我现在判断这条路 ROI 不够。如果你也在做或者考虑做类似项目，希望能给你省一点时间，至少给一个不同视角。

## 一、我想做什么

先把场景说具体。我想做的不是"能跑通一次给老板演示"的 demo，是个能日常用的东西：

> 我在聊天框对它说"用 XX 主题发一条小红书文字笔记"，它自己打开小红书 App、生成内容、填好标题正文、加好话题标签、发布、清后台、回桌面。整个过程我不用碰手机，可以接着写代码或者去倒杯水。

期望的能力边界包括：能稳定连续发 5~10 条不出错；能处理意外（版本更新弹窗、网络抖动、隐私协议提示、UI 偶发变化）；能理解模糊指令（"再发一条类似的"应该自己想起上次的主题方向）；能记住我的偏好（"我喜欢温暖口语化的风格"应该被记下来，下次自动应用）。

为什么这件事看起来值得做：

- **手工痛点是真实的**。运营几个小红书账号的人都知道，发笔记的机械流程让人发疯——每次都是同样几个按钮、同样的位置、同样的等待动画。
- **市面上现有方案都不够**。RPA 类工具（Auto.js、按键精灵）能跑但要写脚本，UI 一改全废，且不智能；Appium 这种自动化测试框架门槛高且只服务测试场景；纯 AI 类目前还都在 demo 阶段。
- **新一代 VLM 让"通用屏幕理解"看起来可行**。GPT-4V、Claude 3.5、Kimi 视觉模型理论上不需要为每个 App 写脚本，看截图就能理解内容做决策。


## 二、技术方案

整体架构 5 层，每一层我都是想清楚再写的，不是堆出来的：

```
对话 UI（Gradio 双栏：左聊天右实时截图）
    ↓
Agent 层（LangGraph 状态机：load_context → route → chat/skill → summarize）
    ↓
Skill 层（注册表机制，每个 App / 任务一个技能模块）
    ↓
Device 层（uiautomator2 封装 + 强制等待 + 互斥锁 + 取消令牌）
    ↓
真机（USB 连接的安卓手机，atx-agent 跑在手机上）
```

**感知层**用 UI 树 dump（uiautomator2 的 `dump_hierarchy`）+ 截图。两者并用：UI 树给精确像素 bounds，截图给视觉上下文。后面我会讲到，这两路信号哪一路都不够，必须组合。

**决策层**用 Kimi 视觉模型。国产模型 API 文档和实际可用列表经常对不上，一定要先写个 verify 脚本试一遍再写正式代码。

**执行层**：触屏 / 滑动 / 系统键全部走 uiautomator2 → adb 注入；中文输入坑大一些——`adb shell input text` 不支持中文，必须装 ADBKeyboard.apk 用广播协议注入。OPPO 手机装 APK 还遇到 "[-99]" 安全拦截，需要去开发者选项手动开"通过 USB 安装"。

**Agent 编排**用 LangGraph。状态机化的 5 节点：load_context（从 SQLite 拉历史和事实）、route（让 Kimi 文本模型判断意图，输出 JSON）、chat_reply（闲聊分支）、run_skill（技能调用分支）、summarize（写记忆 + 抽取事实）。

**记忆**用 SQLite，4 张表——会话、任务、用户事实、技能元数据。事实抽取的实现是：每轮对话结束后让 LLM 看一段固定 prompt 判断"这句话里有没有可以长期保存的稳定事实"，有就 upsert。我跑过一次，用户说"以后用沉稳实干的语气和我说话"，自动抽取出 `communication_style=沉稳实干` 存进了 facts 表。这部分工作得意外的好。

**几个关键设计决策**，每一个我后面都付出了代价才学到：

1. **VLM 输出语义动作而非像素坐标**。一开始让模型输出 `{action: tap, x: 540, y: 927}`，模型把红色加号坐标估到 (540, 1268)，差了几百像素，连续 5 次都点错。后来改成 `{action: tap_id, id: 56}`——模型从带序号的 UI 元素列表里挑序号，代码查表给精确坐标。这个改动是我做的最有效的一次重构，点击精度提升很大。

2. **状态机 + 出场守卫**。模型会"假装"完成了一个状态然后跳到下一个。比如还没真正输入正文就声称"FILL_BODY 完成，转 ADD_TAGS"。我给每个状态加了出场守卫：`FILL_BODY` 必须有累计 100 字以上的 `type_text` 调用才能推进，否则驳回 + 系统提示"还差 N 字"。

3. **页面变化检测**。模型 tap 完"下一步"后页面没真的跳转（动画没完成或 tap 偏了），但模型坚持声称"已进入下一页"，开始按下一页的逻辑操作。我做了 Jaccard 相似度对比 UI 树前后签名，相似度 > 75% 就撤销状态跳转。但这个机制本身又有 bug——某些"页内交互"（点输入框聚焦）也不会改变 UI 树，会被误判为"页面没变"，然后正确的状态被错误撤销。又加了"只对导航类动作做验证"的规则才修好。

写到这你可能已经看出来：这套架构里"防 AI 犯傻"的代码占了一半以上。这个比例一直让我不舒服，但当时还没看清根因。

## 三、取得的初步效果

诚实点说：MVP 跑通了，但远远谈不上稳定。

**能做的事**：
- 端到端发布：自然语言 → 打开小红书 → 生成内容 → 进入编辑页 → 填标题正文标签 → 点发布 → 看到"发布成功" → 回桌面。能完整跑通的成功 case 我手上有 5~6 次。
- 闲聊 + 记忆：能正常对话，能记住稳定偏好。这部分相对稳。
- 浏览技能：能刷推荐流，进入笔记详情页，把内容用一句话总结上报回结果列表。

**不能稳定做的事**：
- 同样的 prompt 跑 10 次，能完整发出去的大概 4~5 次。其他要么卡在某一步死循环（模型反复点同一个元素），要么发完后又开始第二条（不知道任务结束了，最离谱的一次发了两条一模一样的笔记），要么正文压根没输入就跳到加标签了。
- 平均每次完整发布 25~40 个 VLM 调用，单次任务耗时 4~8 分钟。对"日常用"来说这个成本太重——我手动 30 秒搞定的事，给 AI 4 分钟还经常失败，倒贴钱。

**稳定能复现的条件**：手机解锁状态、小红书已登录、网络稳定、当天没有版本更新弹窗、UI 树足够清晰。任何一个条件不满足就大概率失败。

我没有录 demo 视频。坦白讲，给别人看的成功 case 是我跑了好几次后挑出来最干净的一次，不是日常状态。

## 四、达不到预期的三个具体差距

这一节是我整篇文章里写得最仔细的，因为这些细节就是我学到的全部东西。

### 4.1 点击精度：模型既看不准也猜不对

这是我花时间最多的问题。

**典型失败 case**：发布任务的第一步是点底部红色加号。这个按钮在 uiautomator2 dump 的 UI 树里 content-desc 是「发布」，bounds 在屏幕底部正中（cx=540, cy=2208）。我让模型从带序号的元素列表里选，它选中了 `#60 '市集' (324, 2209)`——把旁边的"市集"图标当成了加号，连续好几步都点错。

为什么模型会犯这种错？我事后想，至少两个原因：
1. **它"知道"加号在底部正中**，所以倾向于选 cy 接近 2200 的元素，但底部那一排有 5 个图标（首页、商城、加号、消息、我），它分不清哪个才是中间那个。
2. **它"看到"了红色加号，但 UI 树里没有颜色信息**——文本签名（"市集" / "发布" / "我"）和视觉信号（哪个是红色按钮）没对齐。

我尝试过几种优化：
- **元素列表加 X 坐标排序**，让模型看到 #56 cx=540 是最居中的。改善有限，模型还是经常选邻近的。
- **加位置过滤器**（"必须 cx 在屏幕宽度 0.4~0.6 之间"）。这就是硬编码了，违背了"模型自主决策"的初衷，等于回到 RPA。
- **降级到 tap_coords 兜底**。模型真找不到元素时让它直接出坐标，但坐标精度同样很差，经常偏 100~200 像素。

**另一个让我崩溃的 case**：模型 tap 了"添加标题"输入框（这是个空的 EditText，placeholder 显示"添加标题"），下一步它需要 type_text 输入标题文字。但模型看 UI 树里"添加标题"还在（占位文字未消失，因为输入框为空），就以为"啊我没点中"，又 tap 一次，再 tap 一次……连续 5 次都在点同一个输入框，从不输入。我加的死循环检测把任务杀了。

我加了一个"上一步 tap 过的元素如果在新 UI 树里还在 → 注入系统提示『不要再 tap 了，立刻 type_text』"。部分缓解。但这个就是在用规则补模型的常识缺失——常人都知道点完输入框下一步是输入，模型不知道，要写代码告诉它。补到这里我已经开始怀疑这条路了。

### 4.2 运行速度：每一步都很慢，加起来就是噩梦

一次完整发布的耗时分布（粗略统计）：

| 环节 | 单次耗时 | 占比 |
|---|---|---|
| 截图 + UI 树 dump | 0.5 秒 | 5% |
| Kimi 视觉模型调用 | 3~5 秒 | **70%** |
| 执行动作 + 强制等待 | 1~2 秒 | 20% |
| JSON 解析 + 状态更新 | <0.5 秒 | 5% |

瓶颈非常明确：模型调用。Kimi 视觉模型的端到端延迟稳定在 3~5 秒，30 步就是 90~150 秒，加上 30 步执行约 60 秒，总共 3~4 分钟。中途若有重试或死循环，5~8 分钟。

我尝试过的优化：
- **降到 8k 模型省钱省时**：上下文长度不够，路由 prompt 截断，决策质量明显下降。
- **缓存 UI 树**（同一页面内多个动作只 dump 一次）：节省 0.3 秒/步，杯水车薪。
- **planner / executor 双层（参考 UFO 思路）**：planner 用便宜的文本模型出"下一步要进哪个状态"，executor 用视觉模型出具体动作。理论上能省 30% 视觉调用，但实际上 planner 自己的判断也不一定准，反而拉长了链路。我没把这套完整跑通就放弃了。

模型本身的速度是硬约束。除非换更快的模型（CogAgent 本地 GPU 部署？专用 GUI 模型 API？）或者把决策粒度做得更粗（一次决策驱动多个原子动作），否则没法显著改善。这两个方向都需要不小的工程投入。

### 4.3 智能程度：长 horizon 任务必崩

单步操作模型还行，做完整任务必崩。崩的姿势主要三种：

**姿势一：忘记任务已结束**。发完笔记看到"发布成功"后，模型该 press_key HOME 然后 done，但它经常觉得"任务还没完"，又开始去找加号准备发第二条。我加了状态机的 DONE 状态强制退出，但模型经常在 VERIFY → CLEANUP 之间反复横跳。最离谱的一次它真的发了第二条一模一样的笔记，幸好我在旁边手动停了。

**姿势二：跳过中间步骤**。流程是"输入标题 → 输入正文 → 加标签 → 发布"，模型 tap 完正文输入框后，下一步直接打 `#标签1` —— 把"输入正文"完整地跳过了。它的 thought 里写"正文已经输入完成，接下来加标签"，纯纯的幻觉。我加了出场守卫（FILL_BODY 必须 type 100 字以上才能推进）才挡住，但这又是用工程补能力。

**姿势三：遇到意外完全不会处理**。XHS 偶尔弹版本更新提示、隐私政策弹窗、运营活动 banner，模型的反应像抽奖——有时会关掉、有时去点弹窗里的按钮、有时直接懵（回一个 wait 然后什么都不做）。这类"非预期界面"的处理需要模型有很强的常识 + 灵活的规划能力，目前的视觉模型差得远。

这些问题不是改几条 prompt 能解的。我尝试过：
- **在 prompt 里枚举常见弹窗类型**：列了 5 种，第 6 种出现照样懵。
- **加 fallback：连续 3 步无进展 → 让模型重新规划**：重新规划出来的方案常常更糟。
- **思维链强化**（每步先 thought 再 action 再反思）：token 成本暴涨，效果改善很小。

**核心感受**：模型在"原子操作"层面已经挺聪明，但在"任务级状态跟踪 + 错误恢复"层面跟我们想要的差太远。我加的所有规则、状态机、出场守卫、页面校验，本质上都是在用工程手段补模型能力的缺口。补到最后我发现一件让我很不爽的事：**我做的越来越像 RPA 框架，越来越不像 AI Agent**。

## 五、转向 Mac，意识到问题不在 AI

写到这里我已经有点泄气。但我没有立刻放弃，做了一次"控制变量实验"。

我猜想：手机的问题可能根源在 Android 平台本身。安卓碎片化严重（厂商定制 ROM 各种自绘），AccessibilityService 元素质量参差不齐（小红书的某些按钮根本不在 UI 树里出现），手机屏幕 1080×2400 信息密度高、按钮小、误触代价大。所以我转去 Mac 试了一段时间，主要用 Anthropic 的 cua-driver（Computer Use Agent driver）。

**Mac 上看到的现象，对比强烈**：

- **原生 app 效果好得吓人**。我让它打开备忘录写一条笔记、用 Finder 找特定文件、用 Mail 写邮件、在 System Preferences 里改设置——全都行云流水。AX（Accessibility）树里每个按钮、菜单项、文本框都精确标注了角色（AXButton / AXTextField / AXMenuItem）和可访问标签，模型基本不用猜坐标，直接按角色和标签选就行。
- **第三方 app 立刻拉胯**。Slack、Notion、各种 Electron 套壳的 app，AX 树里大片是 `AXGroup`+`AXUnknown`+`AXGenericElement`，模型根本不知道屏幕上有什么。我让它在 Slack 里给某个频道发一条消息，它点了三次都点偏，最后跑去点了别人的头像把人家个人资料打开了。

这两端经历对比让我突然想通了一件事：

> **AI 操作设备这件事，限制根本不是 AI 模型有多强，是主流操作系统和应用生态从来没为 AI 设计过。**

苹果的 AX 框架本来是给残障用户的辅助技术设计的，不是给 AI 设计的。原生 app 因为受平台规范约束、且开发者重视无障碍设计，AX 信息质量好，AI 顺带受益。但绝大多数现代第三方 app（特别是用 Electron / React Native / Flutter / 自绘引擎写的）压根不实现完整的 AX 接口，AI 看屏幕就只能看像素瞎猜。

Android 上类似——uiautomator2 拿到的 UI 树质量高度依赖 app 是不是规范实现了 native View。小红书这种重度自绘 + 自定义 View 的 app，UI 树里丢失大量结构信息，AI 看到的远不是"完整的屏幕状态"。

**模型再强，看到的世界本身是不全的，再聪明也变不出信息。**

意识到这件事的那个晚上我其实有点恍惚——我之前花的精力，本质上都是在帮模型"猜"那些应该被 app 主动告诉它的东西。这不是 AI 的活，这是基础设施的活。

## 六、我对 GUI Agent 这条技术路线的判断

基于上面的经历，我形成了一个还不太敢百分百确定但越来越坚定的判断：

**GUI agent（让 AI 像人一样看屏幕点按钮）有结构性的天花板。这个天花板不是模型能力，而是它要求 AI 单方面去适应一个不为它设计的世界。**

为什么我说是"结构性"的：

1. **应用生态不会为 AI 大规模改造**。商业 app 的开发优先级是用户增长、留存、付费转化，不是让 AI 看得清。期待小红书、抖音、淘宝主动去为 AI agent 优化 AX 树，约等于零。微信连"消息撤回"都改了 N 年才改顺，让它给 AI 暴露完整无障碍信息？不太可能。

2. **新技术栈在加剧问题**。Web 应用越来越喜欢 canvas + WebGL（Figma、Notion、各种数据可视化），桌面应用越来越喜欢 Electron 套壳，移动应用越来越喜欢 Flutter / 跨端框架——这些技术栈对原生 accessibility 都不友好。屏幕越来越像一片"像素海洋"，对 AI 来说越来越不可读。

3. **模型变强不能解决信号缺失问题**。再强的 VLM，输入一张全 canvas 渲染的截图，识别精度也难低于人工写规则。这是信息论问题，不是建模问题。

业界其实已经在分叉，两条路径同时在跑：

**路径 A：硬刚像素 → 动作的端到端映射**。Anthropic Computer Use、OpenAI Operator、CogAgent、OS-Atlas 等专用 GUI 模型都在赌这条。短期一定会越来越好，因为模型本身在变强、训练数据在变多。但前面说的天花板存在。

**路径 B：绕开 GUI 走协议**。MCP（Model Context Protocol，Anthropic 推的）、function calling、A2A（agent-to-agent）通信。让 app 主动暴露能力给 agent，AI 不需要"看屏幕"，直接调"发笔记(标题, 正文, 标签)"这种结构化接口。

我自己倾向于 B，理由很朴素：

- A 路径要求改变 AI（让它能看懂任意像素海洋），但改变 AI 的边际成本（更大的模型、更多的训练数据、更强的硬件）越来越高。
- B 路径要求改变 app 生态（让它暴露能力），但 B 路径有一个 A 没有的东西——**正向激励**。如果我是 app 开发者，主动暴露 MCP 接口意味着我的服务能被 ChatGPT、Claude 等 AI 入口调用，相当于多了一个分发渠道。这是个商业决策，会自然发生。GUI agent 的 ROI 反向：app 开发者优化 AX 树能换来什么？没什么。所以不会发生。

我不是说 GUI agent 完全没用——做特定 app 的内部自动化仍有明确价值，且边界可控。但作为"通用智能体"的一个核心入口，它不会赢。

我猜未来 3~5 年的格局可能是：基础模型 + MCP 协议层成为赢家，GUI agent 退守到"协议未覆盖的长尾场景"，现在大量 GUI agent 项目会和当年大量 RPA 工具一样，沉到企业内部成为某些团队的小工具，但不会有 toC 的爆款。

## 七、为什么放弃，以及我学到了什么

基于上面的判断，我决定不在 GUI agent 方向继续投入。

要明确一点：**我不是因为做不下去而放弃**。最后一版代码已经能端到端跑通了，再花两周大概能把单技能成功率从 50% 提到 70%。我放弃是因为我判断 ROI 不够——继续投入的边际收益（提高单技能稳定性）远小于这条路本身的天花板。区分这两个我觉得很重要：**前者是失败，后者是止损**。

这两个月我学到的东西，列五条：

1. **架构决策比模型调优更值钱**。我做的最有效的两个改动是 SoM-lite和状态机出场守卫（防模型谎报完成）。两个本质都是"把模型不擅长的事代码化、把模型擅长的事留给模型"。这种"模型/代码边界感"是 agent 工程最核心的 skill，不在任何教程里，要踩坑才能长出来。

2. **不要用 prompt 补能力缺口**。每当我发现模型某个行为不对，第一反应是改 prompt。改了十几轮才意识到：**如果一个能力缺口需要靠 prompt 反复强调才能勉强工作，这个缺口大概率是模型本身的，prompt 改不动**。该上代码就上代码，该换模型就换模型，该换路径就换路径。继续改 prompt 只是在自我安慰。

3. **要早做"控制变量实验"**。我在 Android 上耗了一个多月才转 Mac 做对照。如果第二周就同步在 Mac 上做对照实验，会更早看清"这是平台/生态的问题不是 AI 的问题"，少走至少三周冤枉路。

4. **承认天花板比突破天花板更省力**。我可以再花两个月把 Android 端的稳定性做上去，但那对世界没有边际贡献——这条路别人也在走，且终究撞墙。承认天花板意味着可以把精力转到 MCP / 协议层 / 端侧模型这种还有结构性增长空间的方向。

5. **先想清楚"用户每天会用吗"再写一行代码**。我做这个 agent 的过程中其实已经在心里给它打了"我自己不会每天用"的标签——4 分钟发一条笔记还经常失败，我宁愿手动 30 秒。这个信号其实第一个月就有了，但我没敢直面。下次做项目第一周就要回答这个问题。

## 结尾

接下来我打算花点时间研究 MCP 生态。不是说一定会做这个方向，但我想看看离我"能 AI-native 调用日常软件"这个梦想最近的路是不是它。

有一个不算 takeaway 的 takeaway：花时间做一个最后没继续的项目并不可惜——它给了我对一整条技术路线的判断力。这个判断力比一个能跑的 demo 值钱得多。
