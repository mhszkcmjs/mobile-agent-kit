## 2. 阶段 A：低成本高回报

### A1. SoM-lite：tap_id 替代 tap_text/tap_desc/tap_coords

**思路**
把当前屏幕的可点击元素全部编号，给 VLM 看的是序号 + 标签 + 坐标。VLM 输出 `{"action":"tap_id","params":{"id":N}}`，代码查表拿到精确坐标后点击。

**改造点**
- `vlm_loop._dump_clickable()`：每个元素加序号 `[#1] / [#2] ...`，并把 `(idx → bounds)` 表存进 `step_context`
- `vlm_loop._execute()`：新增 `tap_id` 分支，根据 idx 查表 + 精确点击；保留 tap_text 作为兜底
- 系统 prompt：把"优先 tap_text/tap_desc"改成"必须用 tap_id；只有元素列表里没合适的才用 tap_coords"

**验收**
- 跑一次发布技能，统计 `tap_text/tap_desc UI 树没找到` 警告数 → 应该接近 0
- 跑 3 次发布，平均步数下降 ≥ 30%

### A2. 状态机化的任务推进

**思路**
技能层维护一个显式状态枚举，VLM 在每步既要给动作也要给"是否进入下一状态"。状态到 DONE 立刻退出循环，不让模型自由发挥。

**状态定义（粗粒度，7 状态）**

| 状态 | 含义 | 退出条件 |
|---|---|---|
| INIT | 任务刚启动 | App 已打开 |
| NAVIGATE | 在 App 内找到目标功能入口 | 进入目标编辑/操作页 |
| INPUT | 主要内容输入/操作 | 该输的输完 |
| ENRICH | 附加内容（标签/媒体等） | 附加完成 |
| SUBMIT | 确认提交 | 服务器接受 |
| VERIFY | 检测成功标志 | 看到关键词 |
| CLEANUP | 回桌面/清后台 | is_on_launcher == True |
| DONE | 终结 | — |

**改造点**
- `vlm_loop.run_vlm_loop()` 新增 `states: list[str]` 参数（可选）
- `AgentState`/`SkillContext` 持当前状态
- VLM 输出加字段 `next_state`
- 收到 `next_state == "DONE"` → 立刻退出循环
- 状态在历史 prompt 里显示给模型

**验收**
- 跑发布技能，状态序列应该是 INIT → NAVIGATE → INPUT → ENRICH → SUBMIT → VERIFY → CLEANUP → DONE，不再重复
- 不再出现"发完一条又开始第二条"的现象

---

### A3. 动作后强校验，砍掉确认型 VLM 调用

**思路**
某些状态转移代码本身可以判定，不让 VLM 再花一步确认。

**确定可代码判定的转移**
- `launch_app` 后：`device.current_package() == 目标包名` → 自动 INIT → NAVIGATE
- `type_text` 后：UI 树里目标输入框非空 → INPUT 进度推进
- `tap "发布"` 后：屏幕出现 "上传中"/"发布中" → SUBMIT → VERIFY

**改造点**
- `vlm_loop._execute()` 在执行后做一次轻量校验
- 校验通过则在历史里追加 `[系统]: 验证通过 X` 提示，下一步 prompt 里 VLM 看到，跳过自我确认
- 不替代 VLM 决策，只是减少冗余检查

**验收**
- 视觉调用次数下降 ≥ 15%

## 3. 阶段 B：架构提升（A 阶段稳定后再做）

### B1. Planner / Executor 双层

- **Planner**（纯文本，便宜）：看目标 + 历史 + 当前状态 → 决定下一状态目标
- **Executor**（视觉）：看屏幕 + 状态目标 → 决定具体动作

### B2. UI 树缓存按页面复用

- 同一屏幕连续做多个动作时（点标题 → 输入文字），缓存上次 dump
- 监测包名变化或显著 UI 变化时刷新
